# PRD: Gateway V2 Enhancements

## Introduction

Five quality-of-life improvements for pi-discord-gateway that make it friendlier to use, more self-maintaining, and more capable. The changes cover: open channel access policy, user-facing abort, archived session cleanup, a scheduled task automation engine, and a built-in file-sending tool for the pi agent.

## Goals

- Eliminate the need to register every guild channel one-by-one
- Let users cancel an in-flight pi invocation from Discord
- Prevent archived sessions from accumulating indefinitely on disk
- Provide a complete scheduled/recurring task subsystem usable by pi via CLI
- Give pi a native CLI tool to send files (and text) to Discord channels

## User Stories

### US-001: Channel access policy — config, DB, and auto-register logic
**Description:** As a gateway operator, I want to choose an open, open-trigger, or allowlist channel policy at setup time (and via config) so that guild channels are automatically available without manual registration.

**Acceptance Criteria:**
- `CHANNEL_POLICY` env var accepted with values `open`, `open-trigger`, `allowlist` (default `allowlist` for backward compatibility)
- `config.ts` exposes `channelPolicy` field
- `setup.ts` wizard includes a Channel Policy step asking the user to choose one of the three modes
- Config file generated by setup includes `CHANNEL_POLICY` line
- `EXCLUDED_CHANNELS` env var accepted as comma-separated channel IDs; exposed in config
- In `discord.ts` `handleMessage`: when policy is `open` or `open-trigger`, unregistered guild channels are auto-registered on first message (deriving name from `guild.name #channel.name`, folder from `ch_<id>`, `requiresTrigger` from policy)
- Existing allowlist behavior is unchanged when policy = `allowlist`
- Excluded channels are never auto-registered and never responded to
- Typecheck passes
- Tests pass (unit test for the policy-gated auto-register logic)

### US-002: Abort / cancel command
**Description:** As a Discord user, I want a `/pi stop` slash command to abort the currently running pi invocation in my channel and clear queued messages, so I can recover from mistakes quickly.

**Acceptance Criteria:**
- `queue.ts` exports `abortChannelTask(jid): { aborted: boolean; cleared: number }` that aborts the active controller for the channel and clears pending messages
- `slash-commands.ts` registers a new `stop` subcommand on the existing `/pi` command
- When invoked: if channel has an active task, it is aborted; pending messages are cleared; user receives a confirmation message with count of cleared messages
- When invoked with no active task: user receives "No active task in this channel"
- Agent process receives SIGTERM (existing abort path in `agent.ts` is reused)
- Typecheck passes
- Tests pass (unit test for `abortChannelTask`)

### US-003: Archived session auto-cleanup
**Description:** As a gateway operator, I want archived session directories to be automatically deleted after a configurable retention period so disk space is managed.

**Acceptance Criteria:**
- `ARCHIVE_RETENTION_DAYS` env var (default `30`, `0` = never clean)
- `config.ts` exposes `archiveRetentionDays`
- New module or function `cleanupArchivedSessions()` scans `sessionsDir` for `__archived_<timestamp>` directories, parses the timestamp from the directory name, deletes those older than retention
- Cleanup runs on a 24-hour interval timer, started/stopped alongside the gateway lifecycle in `index.ts`
- `piscord archive cleanup [--dry-run]` CLI command for manual trigger
- `piscord archive list` CLI command to list archived sessions with age
- Typecheck passes
- Tests pass (unit test for timestamp parsing and age filtering)

### US-004: Scheduled tasks — DB schema, scheduler engine, and CLI
**Description:** As a pi agent, I want to create, list, enable, disable, and remove scheduled tasks via CLI so the gateway executes prompts on a cron schedule or at a specific time.

**Acceptance Criteria:**
- New `scheduled_tasks` table in DB: id, name, type (`once`/`recurring`), schedule (ISO datetime or cron expression), channel_jid, prompt, enabled, last_run_at, next_run_at, created_at, created_by
- DB functions: `addTask`, `removeTask`, `enableTask`, `disableTask`, `listTasks`, `getTask`, `updateTaskAfterRun`
- New `scheduler.ts` module: checks every 30s for tasks where `enabled = 1 AND next_run_at <= now`; on trigger, enqueues a message into `message_queue` with sender_name `scheduler`; updates `last_run_at` and computes `next_run_at` (for recurring) or disables (for once)
- Cron parsing via a lightweight library (`croner`) or minimal built-in parser
- `MAX_SCHEDULED_CONCURRENCY` config (default `1`) limits how many scheduled tasks run simultaneously (separate from the main concurrency pool)
- Scheduler lifecycle integrated in `index.ts` (start/stop)
- CLI commands:
  - `piscord task add --name <n> --schedule <cron|iso> --channel <jid> --prompt <text> [--once]`
  - `piscord task list`
  - `piscord task remove <id>`
  - `piscord task enable <id>`
  - `piscord task disable <id>`
- Typecheck passes
- Tests pass (unit tests for next-run computation and task lifecycle)

### US-005: Send-file tool — CLI command for pi to send files to Discord
**Description:** As a pi agent, I want a CLI command to send one or more files (with optional text) to a Discord channel, so file delivery is a first-class gateway capability without modifying the response pipeline.

**Acceptance Criteria:**
- `piscord send --channel <jid> --text <message> --file <path> [--file <path> ...]` CLI command
- Implementation uses discord.js `AttachmentBuilder` to upload files
- Requires the gateway daemon to be running (command communicates with the running gateway or uses a lightweight Discord client)
- Approach: a thin IPC mechanism — the CLI writes a send-file request to a known Unix socket or a DB-backed outbox table; the running gateway picks it up and sends via Discord
- Simpler alternative: the CLI command directly creates a minimal Discord REST client using the bot token from config to POST the file, no IPC needed
- File size limits respected (`MAX_ATTACHMENT_BYTES`, max 10 files per message per Discord limit)
- Text is optional; if provided, sent as message content alongside files
- Typecheck passes
- Tests pass (unit test for argument parsing and file validation)

## Functional Requirements

- FR-1: Channel policy system with open / open-trigger / allowlist modes
- FR-2: Excluded-channels list in config
- FR-3: `/pi stop` slash command to abort and clear
- FR-4: Configurable archive retention with automatic cleanup
- FR-5: Archive management CLI (list, cleanup)
- FR-6: Scheduled task storage, scheduling engine, and CLI management
- FR-7: File-sending CLI tool using Discord REST API

## Non-Goals

- No web dashboard or admin panel
- No per-guild policy differentiation (single global policy)
- No task editing after creation (remove + re-add)
- No streaming file output from pi stdout (the CLI tool is a separate invocation)
- No complex RBAC or permission model for scheduled tasks

## Technical Considerations

- All new config vars have sensible defaults and are backward-compatible
- The scheduler must not starve interactive messages — separate concurrency limit
- For the send-file tool, using the bot token directly in a REST call is simplest and avoids IPC complexity
- Cron library: `croner` is ~5KB, zero-dependency, well-tested
- The project uses ESM, strict TypeScript, vitest for testing

## Success Metrics

- Zero manual channel registration needed when policy = open
- Abort latency < 6 seconds (SIGTERM + SIGKILL fallback)
- Archived sessions older than retention are cleaned within 24 hours
- Scheduled tasks fire within 60 seconds of their target time
- File sending works for all Discord-supported file types up to 25 MB
